# UI Design Guide — 구현 전 디자인 정리

## 언제 적용하는가

- 명시적: 사용자가 `/ui-design` 또는 UI 작업을 요청할 때
- 자동 감지: 페이지, 컴포넌트, 모달, 레이아웃 구현 작업이 시작되려 할 때 제안

단순 스타일 수정(색상 변경, 패딩 조절 등)에는 적용하지 않는다.

## 절대 규칙

**UI 코드를 작성하기 전에 반드시 아래 질문을 순서대로 진행하고 사용자 승인을 받는다.**

조금이라도 명확하지 않은 부분이 있으면 추측하지 말고 사용자에게 물어본다.

## Phase 1: 요구사항 정리

한 번에 하나씩, 순서대로 질문한다. 사용자 답변을 받은 후 다음 질문으로 넘어간다.

### 1. 무엇을 만드는가

- 페이지 / 컴포넌트 / 모달 / 레이아웃 중 무엇인지
- 어떤 기능을 수행하는지
- 어디에 위치하는지 (라우트, 부모 컴포넌트 등)

### 2. 디자인 레퍼런스

사용자에게 레퍼런스가 있는지 물어본다:

- **Figma 링크가 있으면**: Figma MCP로 디자인을 가져온다
- **스크린샷/URL이 있으면**: 해당 레퍼런스를 기준으로 진행한다
- **없으면**: 단계적으로 물어본다
  1. 전체 스타일 방향 (미니멀, 대시보드형, 카드 그리드 등)
  2. 레이아웃 구조 (사이드바+콘텐츠, 탭 기반, 스크롤 등)
  3. **참고 서비스 제시** — 사용자가 방향을 잡기 어려우면 구체적 서비스를 예시로 물어본다:
     > "참고할 만한 서비스가 있을까요? 예를 들면:"
     > - Airbnb 스타일 (카드 그리드, 필터 바, 이미지 중심)
     > - Toss 스타일 (미니멀, 여백 넓은, 단계별 플로우)
     > - Notion 스타일 (사이드바 + 콘텐츠, 블록 기반)
     > - Linear 스타일 (다크 테마, 키보드 중심, 리스트 뷰)
     > - 기존 프로젝트 내 다른 페이지
  4. 사용자가 선택하면 해당 서비스의 핵심 UI 특징을 기반으로 구체화

### 3. 반응형 동작

- 모바일/데스크탑 브레이크포인트
- 모바일에서 레이아웃이 어떻게 변하는지 (그리드 → 스택, 사이드바 숨김 등)
- 모달이면: 모바일에서 풀스크린으로 전환되는지
- 특별한 반응형 동작이 있는지 (하단 고정 버튼, 키보드 툴바 등)

### 4. 모션/애니메이션

먼저 물어본다: **이 UI에 애니메이션이 필요한가?**

- **필요 없다**: 넘어간다
- **필요하다**: 구체적으로 물어본다
  - 어떤 요소에 어떤 트랜지션이 필요한지
  - 입장/퇴장 애니메이션 여부
  - 리스트 stagger, 페이지 전환, 호버 피드백 등
  - 기존 프로젝트에 정의된 motion variant가 있으면 그것을 우선 사용

### 5. 접근성 (a11y)

- 키보드 네비게이션이 필요한 인터랙션이 있는지
- 포커스 트랩이 필요한지 (모달, 드롭다운 등)
- aria 라벨이 필요한 커스텀 컨트롤이 있는지
- 스크린 리더 고려사항이 있는지

### 6. 문서화

사용자에게 물어본다: **UI 스펙 문서를 남길 것인지?**

- **남긴다**: 요구사항 정리 결과를 문서로 저장하고 커밋
- **남기지 않는다**: 컨텍스트로 들고 바로 구현으로 넘어간다

## Phase 2: 구현 가이드

요구사항이 정리되면, 프로젝트 컨텍스트를 읽어서 구현 방향을 정리한다.

### 프로젝트 컨텍스트 파악

1. **CLAUDE.md / CONTRIBUTING.md** 읽기 — 컴포넌트 패턴, 폴더 구조, 네이밍 규칙
2. **컴포넌트 라이브러리** 확인 — shadcn, MUI, 자체 디자인 시스템 등
3. **CSS 방식** 확인 — Tailwind, CSS Modules, styled-components 등
4. **i18n 규칙** 확인 — 번역 키 패턴, 네임스페이스
5. **기존 유사 컴포넌트 탐색** — 재사용 가능한 것이 있으면 제안

### 구현 방향 정리

파악한 컨텍스트 + 사용자 요구사항을 기반으로 아래를 정리하여 제시한다:

- **컴포넌트 분리 구조** — 어떤 파일로 나눌지, 프로젝트 패턴에 맞게
- **재사용 컴포넌트** — 기존에 있는 것 중 활용할 것
- **상태 관리** — 로컬 state, 전역 store, server state 중 무엇을 쓸지
- **데이터 흐름** — props, context, fetch 등
- **반응형 구현 방식** — 브레이크포인트, 조건부 렌더링 등
- **모션 구현 방식** — 사용할 라이브러리, variant, 적용 위치
- **접근성 구현** — aria 속성, 포커스 관리 방식

### 승인 후 구현

사용자가 구현 방향을 승인하면 코드 작성을 시작한다.

## 금지 사항

- Phase 1 질문을 건너뛰고 구현하지 않는다
- 디자인이 불명확한 상태에서 추측으로 구현하지 않는다
- 여러 질문을 한 번에 쏟아붓지 않는다 — 하나씩 진행
- 프로젝트 컨벤션을 무시하고 임의 패턴을 사용하지 않는다
- 사용자가 제공한 레퍼런스를 무시하지 않는다
